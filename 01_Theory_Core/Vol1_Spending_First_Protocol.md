### 概要 (Abstract)
本論文は、現代経済学（特に主流派経済学）が抱える致命的な論理的欠陥を、システムエンジニアリングの観点から「バグ」として特定し、現代の管理通貨制度に即した正しい「仕様（Spec）」を再定義するものである。
既存の経済学は、通貨を「金（Gold）」のような有限の実物資産と混同する「商品貨幣論」というレガシーなドライバ上で動作しており、これが「財政破綻論」や「緊縮財政」というシステムエラーを引き起こしている。
本論では、通貨を **「データ（信用記録）」** 、政府を **「システム管理者（Admin）」** 、税を **「ガベージコレクション（GC）」** と定義し直すことで、「スペンディング・ファースト（支出が先）」というプロトコルが、論理的かつ実務的に唯一の正解であることを証明する。

---

### 1. 序論：権限（Permissions）の取り違え
経済停滞の根本原因は、国家運営における「ユーザー権限（User Mode）」と「管理者権限（Admin Mode）」の混同にある。

#### 1.1 家計と政府の非対称性
主流派経済学は、政府の財政を「家計（User）」のアナロジーで語る。家計は収入（Input）がなければ支出（Output）できない。しかし、自国通貨を発行できる統合政府は、システムの「管理者（Admin）」であり、この制約を受けない。

* **User (家計・企業):** 通貨の利用者。予算制約あり。`Input -> Output`
* **Admin (統合政府):** 通貨の発行者。供給制約のみ。`Output -> Input`

管理者が「金がないから機能を追加できない」と嘆くのは、自らがデータベースへの書き込み権限（Insert権限）を持っていることを忘れた滑稽なエラーである。

### 2. 通貨プロトコル：データとしての貨幣

#### 2.1 キーストローク・マネー (Keystroke Money)
現代の管理通貨制度において、貨幣とは物理的な貴金属ではなく、バランスシート上の「情報（Data）」である。政府支出とは、中銀がキーボードを叩き（Keystroke）、民間の銀行口座の数値を「書き換える」操作に他ならない。

#### 2.2 システムのリミッター：インフレ率
信用創造（通貨発行）は、データ生成であるため物理的な発行制約（コスト）は存在しない。しかし、物理的リソース（供給能力）には限界がある。
したがって、財政の制約条件は「税収（予算）」ではなく、 **「悪性インフレ（システム過熱）」** というサーマル・スロットリング（熱暴走制御）が作動するラインまでとなる。

---

### 3. 実行シーケンス：スペンディング・ファースト (Spending First)

「税金が財源である」という主流派のドグマは、システムの時間軸（Timestamp）を逆転させた因果律のバグである。

#### 3.1 処理フローの証明
論理的に、政府が通貨を発行（支出）する前に、民間から通貨を回収（徴税）することは不可能である。「存在しないデータ」は削除できないからだ。

1.  **Creation (支出):** 政府が国債を発行し、民間の口座に通貨データを生成する。 `Money += 100`
2.  **Circulation (流通):** 民間で経済活動が行われる。
3.  **Collection (徴税):** 決算期に、流通している通貨の一部を回収・消去する。 `Money -= 20`

したがって、プロセスは常に **「支出（Output） $\to$ 徴税（Input）」** の順序で実行される。税は財源（Source）ではなく、事後的な処理である。

#### 3.2 税の機能：ガベージコレクション (GC)
では、財源でないならば税の役割とは何か。それはシステム内の不要なメモリ（過剰流動性）を解放し、ヒープ領域を健全に保つための **「ガベージコレクション (Garbage Collection)」** である。
* **インフレ抑制:** 通貨を間引くことで、システム過熱を防ぐ。
* **格差是正:** 特定のプロセス（富裕層）に偏ったリソースを再配分する。
* **流通強制:** 「納税義務」を課すことで、その通貨の利用を強制する（租税貨幣論）。

### 4. 日本経済のデバッグ：コストプッシュとスタグフレーション

#### 4.1 エラーログの解析
現在の日本における物価上昇（インフレ）は、需要過多による「処理落ち」ではなく、供給制約による「ハードウェア故障」である。

* **デマンドプル:** CPU使用率 100% (High Load) $\rightarrow$ 正常な成長痛。
* **コストプッシュ:** **CPUファン故障 (Hardware Failure)** $\rightarrow$ 処理能力低下による発熱。

輸入価格の高騰や減反政策による供給能力の毀損は、システムに対する「サボタージュ（破壊工作）」である。この状態で「緊縮財政（冷却）」を行うことは、ファンが壊れたPCの電源を抜くことに等しく、システムの死（経済死）を招く。

#### 4.2 リカバリープラン：供給能力への投資
正しい処置は、国債発行（通貨供給）によってリソースを調達し、食料・エネルギー・技術といった **「供給能力（ハードウェア）」を修理・増強すること** である。これにより、インフレ制約の上限自体が引き上げられる。

### 5. 悪性コードの特定：消費税 (Consumption Tax)

#### 5.1 トランザクション・ペナルティ
消費税とは、システム内で発生するあらゆるトランザクション（取引）に対し、一律で「罰金」を科すコードである。

```python
# Malicious Code: Consumption Tax
def transaction(value):
    penalty = value * 0.10
    if user.wallet < penalty:
        system.crash() # 倒産・貧困
    return value
```

#### 5.2 システムへの影響
デフレ（不況）下においてこのコードを作動させることは、処理速度が低下しているシステムにさらに「ウェイト（待機時間）」を挿入する行為であり、**「自殺プログラム（Self-Destruct）」** である。
特にインボイス制度は、末端の小規模プロセス（フリーランス・中小企業）に管理コストと増税を強制し、エコシステム（産業基盤）を破壊する「ウイルス」として機能している。

### 6. 結論 (Conclusion)

本論により、経済政策におけるパラダイムシフトが提示された。

1.  **財政:** 「収支均衡（プライマリーバランス）」という無意味なKPIを廃棄し、 **「機能的財政論（インフレ率と雇用）」** を管理指標とする。
2.  **国債:** 自国通貨建て国債におけるデフォルト確率は、システム上 **「Null（存在しない）」** である。
    * *Note: これは「自社サーバー内（円建て）」での処理に限る。「外部API依存（外貨建て債務）」はこの限りではない。*
3.  **税制:** 消費税という「トランザクション阻害要因」を排除（廃止）し、システムのパフォーマンスを最大化する。

経済とは、希少な資源を奪い合うゼロサムゲームではなく、 **「正しい権限管理」と「適切なコード（政策）」によって、供給能力の限界まで最大化可能なエンジニアリング対象** である。
